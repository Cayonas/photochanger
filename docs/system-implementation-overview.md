# 在线图片转换系统实现说明

本文档基于当前版本的 `PhotoChanger` 实际实现整理，说明系统现在：

- 已经具备哪些能力
- 主要功能是怎么实现的
- 当前技术方案的边界在哪里
- 后续应该如何继续演进

## 1. 系统定位

`PhotoChanger` 目前是一个可运行的在线图片转换工具，用户可以在网页上：

1. 上传一张或多张图片
2. 设置输出格式、质量、宽高和 EXIF 保留策略
3. 执行转换
4. 下载单图结果
5. 将成功结果打包成 ZIP 下载

它已经不是单纯的静态页面，而是包含前端交互、后端处理、任务状态和下载链路的完整 MVP。

## 2. 当前已实现能力

### 图片输入与输出

- 输入格式：`JPEG`、`JPG`、`PNG`、`WebP`、`GIF`、`HEIC`、`HEIF`
- 输出格式：`JPEG`、`PNG`、`WebP`、`GIF`

### 转换能力

- 图片格式转换
- 图片质量调节
- 按宽度或高度缩放
- EXIF 保留
- 自动处理图片方向信息

### 使用体验

- 前端多文件队列
- 前端并发处理
- 每个文件独立显示状态
- 整体进度条展示
- 单文件下载
- 批量 ZIP 下载

### 资源控制

- 单文件大小限制为 `50MB`
- 后端总转换并发限制为 `4`
- 单图结果在服务端内存中临时保存
- 单图结果与批量 ZIP 均在 `30` 分钟后过期
- 后端每 `60` 秒自动清理一次过期数据

## 3. 系统整体结构

当前系统可以分成两层理解。

### 前端层

负责：

- 页面展示
- 接收用户上传
- 维护文件队列
- 提交转换参数
- 调用后端 API
- 展示状态、进度和下载入口

前端技术：

- Vue 3
- Vite
- Fetch API
- 原生 CSS

### 后端层

负责：

- 接收上传文件
- 校验格式、大小和 MIME
- 执行图片转换
- 保存任务状态
- 返回单图下载流
- 生成批量 ZIP
- 定时清理过期结果

后端技术：

- FastAPI
- Pillow
- pillow-heif
- Python 标准库 `zipfile`

## 4. 核心处理流程

## 4.1 单图转换流程

当用户点击开始转换后，处理链路如下：

1. 前端将图片和参数打包为 `FormData`
2. 调用 `POST /api/v1/convert`
3. 后端读取上传内容到内存
4. 后端校验文件大小、扩展名和 MIME
5. 后端调用 Pillow 执行转换
6. 转换结果保存在进程内存中的任务字典里
7. 后端返回任务信息和下载地址
8. 前端再请求下载地址获取结果文件

这里最重要的一点是：当前版本的单图结果不会写入本地磁盘，而是以内存形式短期保存，并在下载接口中以流的方式返回。

## 4.2 批量转换流程

当前的批量处理仍然是“前端调度、后端逐个处理”的轻量方案。

流程如下：

1. 用户一次上传多个文件
2. Vue 将这些文件加入前端队列
3. 前端按设定并发数挑选待处理文件
4. 每个文件独立调用一次 `POST /api/v1/convert`
5. 成功结果会在前端被标记为已完成
6. 前端收集所有成功任务的 `task_id`
7. 调用 `POST /api/v1/batches`
8. 后端根据这些任务在本地生成一个临时 ZIP
9. 返回 ZIP 下载地址给前端

这种方案的优点是实现快、结构清晰，适合 MVP；缺点是任务依赖前端页面存在，不是真正的后台异步任务。

## 4.3 单图下载机制

单图下载通过 `GET /api/v1/tasks/{task_id}/result` 完成。

处理方式是：

- 根据 `task_id` 读取内存中的转换结果
- 组装下载文件名和内容类型
- 通过 `StreamingResponse` 把字节流直接返回给浏览器

这意味着：

- 减少了磁盘写入
- 避免单图结果在服务器上长期堆积
- 但在有效期内会占用一定内存

## 4.4 批量 ZIP 机制

批量 ZIP 下载通过 `POST /api/v1/batches` 和 `GET /api/v1/batches/{batch_id}/result` 完成。

实现方式是：

- 后端从已完成任务中取出内存里的图片字节
- 将这些字节写入一个 ZIP 文件
- ZIP 临时存放在 `backend/storage/batches/`
- 用户通过下载接口直接取回这个 ZIP

所以当前系统是“单图结果走内存，批量结果走临时文件”的混合模式。

## 5. 关键实现原理

## 5.1 文件校验

后端在处理前会做基础校验：

- 文件大小是否超过 50MB
- 扩展名是否在支持范围内
- MIME 类型是否在支持范围内
- Pillow 是否能真正识别该图片

这样可以提前拦截无效请求，减少资源浪费。

## 5.2 图片转换

转换时主要做了这些事情：

1. 读取原始字节
2. 打开图片对象
3. 根据 EXIF 修正方向
4. 按目标格式处理色彩模式
5. 按输入参数调整尺寸
6. 设置质量和 EXIF 保存参数
7. 将输出结果写入字节缓冲区

因此系统做的不是“改后缀名”，而是真正重新编码输出图片数据。

## 5.3 并发限制

后端维护了一个进程内的总并发计数器。

规则是：

- 当前最多允许 `4` 个转换请求同时执行
- 超出时直接返回 `429`

这样做的目的，是避免多个大文件同时进入处理流程，导致内存和 CPU 瞬时占用过高。

## 5.4 过期清理

后端启动时会创建一个后台清理循环。

它会：

- 每 60 秒扫描一次任务字典和批次字典
- 删除超过 30 分钟的单图结果记录
- 删除超过 30 分钟的批量 ZIP 文件和批次记录

这个机制是当前版本避免磁盘和内存无限增长的关键补丁。

## 6. 为什么采用当前路线

当前实现是一条很典型的 MVP 技术路线：

- 先把上传、转换、下载主流程打通
- 再补前端队列和 ZIP 批量能力
- 然后补资源限制和自动清理

这样做的价值是：

- 能快速交付可运行版本
- 容易验证产品流程
- 便于后续逐步替换底层设施

如果一开始就引入 Redis、对象存储、数据库、消息队列，工程复杂度会明显上升，不利于当前阶段快速迭代。

## 7. 当前边界与限制

为了准确理解当前版本，也需要明确它还不是什么。

### 不是持久任务系统

- 任务状态存在进程内存中
- 服务重启后任务记录会丢失
- 不适合长期历史查询

### 不是后台异步批处理系统

- 批量任务仍然依赖前端页面调度
- 浏览器关闭后不会继续推进新任务
- 不适合特别大的批次或长时间任务

### 不是完全无状态架构

- 单图结果暂存在内存
- 批量 ZIP 暂存在本地文件系统
- 多实例部署时需要重新设计共享状态和存储

### 还不是生产级存储方案

- 还没有对象存储
- 还没有 CDN
- 还没有跨实例共享下载能力

## 8. 推荐演进方向

如果项目继续推进，比较合理的演进顺序是：

### 第一步：配置化

把以下参数改为环境变量：

- 文件大小限制
- 总并发限制
- 单图结果 TTL
- ZIP TTL
- 清理间隔

### 第二步：异步任务化

引入：

- Redis
- Celery / RQ 等任务队列

目标是让任务不再依赖前端页面持续在线。

### 第三步：对象存储化

引入：

- S3 / OSS / R2 等对象存储

目标是把下载结果从本地磁盘迁出，提高伸缩性。

### 第四步：数据持久化

引入：

- PostgreSQL

用于保存任务、批次、用户和下载记录。

### 第五步：平台能力扩展

继续扩展：

- AVIF 支持
- 用户体系
- API Key
- Webhook
- 计费与配额

## 9. 总结

当前版本最重要的价值，不只是“能转图”，而是已经建立了一个可以继续演进的真实系统雏形：

- 前端交互完整
- 后端处理链路完整
- 下载链路完整
- 资源控制开始成型

尤其是最近这一步，把单图结果改成“内存临时保存 + 流式下载 + 30 分钟清理”，已经明显比早期“结果持续落盘”的实现更稳，更适合作为下一阶段优化的基础。
