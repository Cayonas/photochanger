# 开发计划

本文档基于 [PRD.md](/C:/Users/29486/Desktop/photochanger/PRD.md) 拆解为可以落地执行的版本。

## 1. 目标与范围

MVP 目标是在当前仓库内先交付一个可运行的“上传 -> 设置 -> 转换 -> 下载”闭环，并为后续批量处理、队列和账户体系留出扩展点。

## 2. 技术决策

- 后端先采用 FastAPI，原因是接口搭建快、类型清晰，适合快速验证 PRD 主流程。
- 图片处理先采用 Pillow，优先覆盖 JPEG、PNG、WebP、GIF 的主流场景。
- 前端先采用静态页面实现 MVP，避免在功能验证阶段引入额外构建复杂度。
- 开发环境存储先落在本地磁盘；生产再切换到对象存储。
- 任务状态先用内存管理；Sprint 2 再接 Redis/数据库。

## 3. 分阶段实施

### Phase 0：项目骨架与规范

目标：
- 搭建前后端目录结构
- 明确 API 契约
- 建立基础存储目录与忽略规则

交付：
- `backend/`
- `frontend/`
- `docs/development-plan.md`
- `README.md`

### Phase 1：单文件转换 MVP

目标：
- 支持拖拽/选择上传
- 支持单文件转换与下载
- 支持质量、尺寸、EXIF 开关
- 展示原图/结果图和元数据对比

接口：
- `POST /api/v1/convert`
- `GET /api/v1/tasks/{task_id}`
- `GET /api/v1/tasks/{task_id}/result`

验收标准：
- 本地可上传 JPEG/PNG/WebP/GIF
- 转换后的文件可正常打开
- 参数变化能影响输出体积或尺寸
- 超限或非法文件会返回可读错误

### Phase 2：批量处理与队列

目标：
- 支持多文件队列
- 支持并行/顺序策略
- 支持总体进度
- 支持 ZIP 下载

实现建议：
- 接入 Redis + BullMQ/Celery
- 任务元数据落 PostgreSQL
- 输出文件批量打包

### Phase 3：账号、历史与配置复用

目标：
- 用户登录
- 历史任务列表
- 重用转换参数
- 匿名/登录配额差异化

### Phase 4：格式扩展与性能优化

目标：
- 补齐 HEIC/AVIF
- 引入大文件异步处理
- 优化内存占用与并发
- 对接 CDN / 对象存储

### Phase 5：开放 API 与商业化能力

目标：
- API Key
- Webhook
- 订阅与配额
- 审计与删除策略

## 4. 当前实现策略

本轮直接完成 Phase 0 + Phase 1，优先把系统跑起来，再迭代复杂能力。

## 5. 已知限制

- HEIC/AVIF 需要额外编解码支持，本版暂未默认启用
- 批量、ZIP 和队列尚未接入
- 任务状态存储为内存级，重启服务会丢失
- 暂未接入数据库、对象存储和用户体系
