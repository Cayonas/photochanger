# 开发计划

本文档基于当前仓库实际实现整理，目标是说明：

- 现在已经完成了什么
- 当前技术方案是如何取舍的
- 下一阶段推荐优先做什么

## 1. 当前目标与范围

当前版本已经完成一个可运行的 MVP，覆盖了在线图片转换工具的核心闭环：

- 上传图片
- 设置输出格式、质量、尺寸、EXIF
- 执行单图或批量转换
- 下载单图结果或批量 ZIP

本阶段重点仍然是验证主流程、稳定性和资源控制，而不是做重型后台任务平台。

## 2. 当前实现状态

### Phase 0：项目骨架与基础规范

状态：已完成

已交付：

- 前后端目录结构
- FastAPI 服务入口
- Vue 3 + Vite 前端工程
- 基础文档与开发说明

### Phase 1：单文件转换 MVP

状态：已完成

已交付：

- 单文件上传与转换
- 输出格式切换：`JPEG`、`PNG`、`WebP`、`GIF`
- 尺寸调整
- 质量控制
- EXIF 保留开关
- 单文件结果下载

接口：

- `POST /api/v1/convert`
- `GET /api/v1/tasks/{task_id}`
- `GET /api/v1/tasks/{task_id}/result`

### Phase 2：批量处理与前端队列

状态：已完成当前版本

已交付：

- 前端多文件队列
- 前端并发调度
- 单文件结果预览与逐个下载
- 批量成功结果 ZIP 打包下载
- 队列状态与整体进度展示

接口：

- `POST /api/v1/convert`
- `POST /api/v1/batches`
- `GET /api/v1/batches/{batch_id}`
- `GET /api/v1/batches/{batch_id}/result`

### Phase 2.5：资源占用控制

状态：已完成

这是当前版本相较于早期实现最关键的更新：

- 单文件转换结果不再落盘，改为保存在内存中并通过流式下载返回
- 批量 ZIP 仍会生成临时文件，但只存放在 `backend/storage/batches/`
- 单图结果和批量 ZIP 均设置 30 分钟有效期
- 后端每 60 秒自动清理一次过期残留
- 单文件上传大小限制保留为 `50MB`
- 后端总转换并发限制为 `4`

这样做的目标是减少服务器本地磁盘长期增长风险，并用并发上限控制内存压力。

## 3. 当前技术决策

### 后端

- 使用 FastAPI 提供接口与静态站点入口
- 使用 Pillow 处理格式转换、尺寸调整和 EXIF
- 使用 `pillow-heif` 支持 HEIC/HEIF
- 使用内存保存短期单图结果
- 使用本地临时文件保存批量 ZIP
- 使用进程内字典保存任务与批次状态

### 前端

- 使用 Vue 3 管理上传队列和页面状态
- 使用 Vite 构建与开发
- 使用浏览器 `fetch` + `FormData` 提交图片和参数
- 由前端负责轻量批量调度与进度展示

### 为什么当前先这样做

- 实现简单，便于继续迭代
- 适合 MVP 阶段快速验证完整业务流程
- 当前需求规模下，复杂基础设施还不是第一优先级
- 已经为后续迁移到对象存储、异步队列、数据库留出了演进空间

## 4. 当前已知限制

- 任务与批次状态保存在内存中，服务重启后会丢失
- 批量任务仍然是“前端驱动”，不是真正的后台异步任务
- 单图结果虽然不落盘，但仍会在有效期内占用服务器内存
- 批量 ZIP 仍然依赖本地磁盘临时保存
- 当前并发控制是进程内级别，不适用于多实例横向扩展
- 还没有用户体系、历史任务中心、计费或 API Key

## 5. 下一阶段建议优先级

### 优先级 1：把关键配置外置

建议先把以下参数改成环境变量：

- 单文件大小限制
- 总并发限制
- 单图结果 TTL
- 批量 ZIP TTL
- 清理间隔

这样部署时更容易按机器配置调整。

### 优先级 2：引入真正的后端异步任务

建议方向：

- Redis
- Celery 或 RQ

目标：

- 用户关闭页面后任务仍可继续
- 更适合长任务和大批量任务
- 为 webhook / 通知机制做准备

### 优先级 3：迁移到对象存储

建议方向：

- Amazon S3
- Cloudflare R2
- 阿里云 OSS

目标：

- 进一步降低本地磁盘风险
- 提高下载链路稳定性
- 为 CDN 加速和多实例部署做准备

### 优先级 4：引入数据库

建议方向：

- PostgreSQL

用于保存：

- 用户信息
- 任务记录
- 批次记录
- 下载记录
- 配置模板

### 优先级 5：扩展格式与平台能力

可继续推进：

- AVIF 支持
- API Key
- Webhook
- 历史任务中心
- 订阅与配额

## 6. 当前阶段总结

当前仓库已经不只是原型，而是一个可运行、可演示、可继续迭代的图片转换 MVP。

现阶段的核心成果有两个：

- 业务闭环已经打通
- 资源控制比早期版本更稳，至少已经避免了“单图结果无限落盘”的问题

接下来最值得做的不是继续堆新功能，而是把“配置化、异步化、存储外置”这三件事逐步补齐。
